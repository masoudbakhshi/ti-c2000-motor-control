# Blink LED — TMS320F28388D controlCARD

A minimal hardware verification project for the TMDSCNCD28388D controlCARD mounted on the TMDSHSECDOCK docking station. The onboard user LED (D1) blinks at 1 Hz, confirming that the clock tree, GPIO subsystem, and JTAG debug connection are all operational before any application firmware is brought up.

---

## Hardware

| Item | Details |
|---|---|
| controlCARD | TMDSCNCD28388D Rev.B |
| Device | TMS320F28388D (F2838x, C28x + CM4 + CLA) |
| Docking station | TMDSHSECDOCK |
| LED D1 | GPIO31 — HSEC connector, pin 82 |
| LED D2 | GPIO34 — HSEC connector (not used in this project) |
| LED logic | Active-low (GPIO LOW → LED on) |

## Project Structure

```
blink_led/
├── main.c               Source — initialization sequence and blink loop
├── blink_led.syscfg     SysConfig — board pin-mux and GPIO direction
└── CCS/
    └── blink_led.projectspec    CCS import descriptor
```

SysConfig processes `blink_led.syscfg` at build time and generates the following files into the build output directory. They are not checked into source control because they are derived artifacts:

- `board.h` / `board.c` — GPIO pin definitions and `Board_init()`
- `device_cmd.cmd` — device-level linker command file
- `board.cmd.genlibs` — library list for the linker

## Getting Started

### 1. Import the project into CCS

1. Open Code Composer Studio and select your workspace.
2. Go to **File → Import → C/C++ → CCS Projects**.
3. Click **Browse** and navigate to the `blink_led/CCS/` folder.
4. Select `blink_led` and click **Finish**.

CCS will copy the source files and register the SysConfig build step automatically.

### 2. Verify the C2000Ware path

Open the project's **Properties → Build → Path Variables** and confirm that `C2000WARE_ROOT` points to your local C2000Ware installation:

```
C:/ti/c2000/C2000Ware_26_01_00_00
```

If you installed C2000Ware to a different directory, update the path variable before building.

### 3. Build

Select the `CPU1_RAM` configuration from the build target dropdown and click the hammer icon (or press **Ctrl+B**). The build will:

1. Run SysConfig on `blink_led.syscfg` → generates `board.h`, `board.c`, and linker files.
2. Compile `main.c` and the generated board source.
3. Link everything into `blink_led.out`.

A clean build should produce zero errors and zero warnings.

### 4. Debug and run

1. Connect the TMDSHSECDOCK to your PC via USB (XDS110 on-board emulator).
2. Click **Run → Debug** (or press **F11**).
3. CCS will flash the `.out` image to SARAM and halt at `main()`.
4. Press **F8** (Resume) to run.

LED D1 on the controlCARD will begin blinking at 1 Hz — 500 ms on, 500 ms off.

## Build Configurations

| Configuration | Code location | Intended use |
|---|---|---|
| `CPU1_RAM` | On-chip SARAM | Development and debug — fast reflash, no Flash wear |
| `CPU1_FLASH` | Internal Flash sector 1–4 | Standalone deployment — persists after power cycle |

For Flash builds, switch the active configuration to `CPU1_FLASH` before building. The linker will place `.text` in Flash and copy `.ramfunc` sections to RAMLS0 at startup.

## How It Works

The F28388D resets with the PLL disabled. `Device_init()` starts the oscillator, locks the PLL to the configured SYSCLK frequency, and powers up the peripheral clock gates. Only after that does `Device_initGPIO()` release the GPIO pin-mux from its reset-state lock.

`Board_init()` — generated by SysConfig from `blink_led.syscfg` — configures GPIO31 as a push-pull output with the initial pin state set high (LED off). The main loop then drives the pin low for 500 ms and high for 500 ms using the `DEVICE_DELAY_US()` busy-wait macro, which is calibrated to SYSCLK at device initialization time.

The active-low LED circuit is standard on TI controlCARDs: the GPIO drives the anode of a current-limiting resistor that feeds into the LED cathode, with the LED anode connected to 3.3 V. Driving the GPIO low completes the circuit; driving it high opens it.

## Extending This Project

This project is intentionally minimal. Common next steps:

- **Add a second LED**: Uncomment or replicate the GPIO write calls for GPIO34 (LED D2) and add it to the SysConfig LED module.
- **Use a timer interrupt**: Replace the blocking `DEVICE_DELAY_US()` delay with a CPU timer ISR to free the main loop for other work.
- **Add a ePWM breath effect**: Drive the LED through an ePWM channel to vary brightness using duty-cycle modulation.
